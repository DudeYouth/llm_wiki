## 需求背景
- 现有规则要求“页面必须关联、避免孤立”，但仅靠 Markdown 双链无法做完整性校验。
- 引入 Graphify 作为关系层，把约束升级为可计算结构，减少链接遗漏。

## 技术选型
- 展示层：Markdown + Obsidian 双链，保持原工作流不变。
- 关系层：Graphify 图谱，存储实体与关系，支持查询与校验。
- 元数据：页面 frontmatter 增加 `graph_id / entity_type / relations / source_ref`，便于双向一致。

## 实体模型（最小集合）
- Solution：执行方案页（wiki/solutions/*）
- Tech：技术知识页（wiki/tech/*）
- Task：任务页（wiki/tasks/*）
- Source：原始方案摘要页（wiki/sources/*）
- GlossaryTerm：术语页（wiki/glossary.md 拆分项）

## 关系模型（最小集合）
- source_summarizes：Source -> Solution
- solution_uses：Solution -> Tech
- solution_has_task：Solution -> Task
- task_depends_on：Task -> Task
- term_described_in：GlossaryTerm -> Solution
- tech_related_to：Tech -> Tech

## 任务拆分
- T1：在 SKILL.md 新增「Graphify 知识图谱规则」章节（目标 / 实体模型 / 关系模型 / 入库时机 / 最小验收标准）。
- T2：在导入工作流的「步骤6：更新全局索引和概览」之后追加「步骤6.5：更新 Graphify 图谱」。
- T3：要求 6.5 的校验结果写入 wiki/log.md。

## 风险与应对
- 风险：一次性引入实体/关系太多，难以落地。应对：先固定 5 类实体 + 6 类关系。
- 风险：Source 页可能短暂孤立。应对：允许 24h 宽限，过期视为不合规。
- 风险：图谱与 Markdown 双链漂移。应对：每次 upsert 同步双链与图谱，删除/重命名联动清理。

## 验收标准
- SKILL.md 可见 Graphify 新章节与「步骤6.5」。
- raw/ 原始约束未被破坏（仍只读）。
- 现有「步骤1~7」语义不变，仅做追加，不做删改。
- 文档风格保持中文 + 短条目，整体不超过 30 行新增。
