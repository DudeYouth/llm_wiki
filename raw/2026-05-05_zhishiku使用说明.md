# zhishiku 知识库使用说明

## 本技能解决什么问题

当你用 Plan 模式产出一个「执行方案」（架构设计、技术选型、任务拆分等），这个技能让 Agent 自动把方案落盘 `raw/` → 校验 → 经 MCP 编译成 wiki 页面 → 同步到 Graphify 图谱 → 记入操作日志。整个过程**无需人类手动写 wiki 或维护索引**。

## 前置条件

- Cursor 已连接名为 `llmwiki` 的 MCP 服务（`--root` 指向本仓库）
- 仓库已启用 Plan Gate 门禁（`.cursor/rules/plan-gate.mdc`）
- Graphify 图谱功能可用（`.cursor/rules/graphify.mdc`）

## 使用流程（简版）

### Step 1: Plan 模式落盘方案

方案产出后，按 Plan Gate 要求落地：

- **方式 A（推荐）**：在 `raw/plan-log.md` 末尾新增一节，节首 fenced `yaml` 含 `status: approved` 或 `ready_for_execute: true`
- **方式 B**：创建独立文件 `raw/YYYY-MM-DD_方案关键词.md`，frontmatter 同上

文件须含：需求背景、技术选型、任务拆分、风险应对、验收标准

### Step 2: 编译进 wiki

调用 llmwiki MCP 的两个工具，**按顺序**执行：

1. `ingest_source` — 参数 `source` 为 `raw/YYYY-MM-DD_*.md` 的**绝对路径**（Windows 含盘符）
2. `compile_wiki` — 无参，增量编译管线

> **禁止**：用 Agent 手写多页 wiki 替代 compile_wiki 产出；用 Shell CLI 绕过 MCP 做编译

### Step 3: 编译后验收

编译完成后，按需执行：

- `wiki_status` — 看全库总览（页数、源变更、上次编译时间）
- `lint_wiki` — 规则质检（坏链、孤儿等）

### Step 4: 图谱同步

Agent 自动对新增或变更的 wiki 页面执行 Graphify upsert（实体与关系），Markdown 双链与图谱保持一致。

### Step 5: 记录日志

在 `wiki/log.md` 末尾追加一行：

```
时间戳 | raw/YYYY-MM-DD_方案关键词.md | updated_pages | added_nodes | added_edges | validation | status
```

示例：

```
2026-05-01T21:20:00+08:00 | raw/2026-05-01_示例方案.md | 6 | 9 | 14 | pass | success
```

- `validation` 取值：`pass`、`warn`、`fail`
- `status` 取值：`success`、`partial_success`、`failed`
- `updated_pages`、`added_nodes`、`added_edges` 为 0 或正整数

## 常见场景

| 场景 | 做法 |
|------|------|
| 只想登记源、暂不编译 | 只调 `ingest_source`，在 `wiki/log.md` 说明原因 |
| 多节方案都在 `plan-log.md` | 应先弄出独立 `raw/YYYY-MM-DD_*.md` 再 ingest |
| MCP 服务不可用 | 在 `wiki/log.md` 记因后按用户指定方式降级 |
| compile_wiki 部分失败 | 仍按规范写入日志，`status` 如实填 `partial_success` |

## 不应做的操作

- 不要手动修改 `raw/` 下的任何文件（只读）
- 不要手写多页 wiki 替代 compile_wiki
- 不要落盘 raw 后跳过 MCP 编译就声称 wiki 已更新
- 不要以 `plan-log.md` 全文作为 ingest_source 目标
- 不要用 Shell CLI 替代已连接的 MCP 执行编译
